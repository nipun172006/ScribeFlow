# Viva Notes

This document provides a student-friendly overview of ScribeFlow's architecture, design decisions, and technical implementations to help prepare for viva and final presentations.

## Architecture Overview

### Frontend (React/Vite)

The frontend is built with React, Vite, and Tailwind CSS. It is responsible for user interaction, file selection, and rendering meeting data. It uploads audio directly to Supabase Storage using the TUS protocol via temporary signed tokens generated by the backend. It never talks directly to Deepgram or Gemini.

### Backend (Express/Node.js)

The backend acts as a secure orchestrator. It holds all the secret API keys (Supabase Service Role, Deepgram, Gemini). It validates requests, signs upload tokens, initiates AI processing pipelines, and safely persists structured data into the database.

### Database & Storage (Supabase/PostgreSQL)

Supabase provides the PostgreSQL database and private S3-compatible object storage. The database uses relational tables to store meetings, transcripts, and action items, and utilizes `pgvector` for semantic search embeddings.

## Why These Technologies?

### Why Deepgram?

Deepgram was chosen for transcription because it offers high accuracy, extremely fast processing (Nova-3 model), and built-in speaker diarisation (identifying _who_ spoke _when_). It processes audio directly from secure, short-lived Supabase download URLs generated by our backend.

### Why Gemini?

Google Gemini was chosen for meeting analysis because of its native support for **Structured Outputs**. By enforcing a strict JSON schema, we guarantee the model returns exact arrays for `executiveOverview`, `keyDecisions`, and `actionItems`, drastically reducing parsing errors compared to standard LLM text outputs.

## Transcription & Extraction

### How Diarisation Works

Speaker diarisation answers "who spoke when?" Deepgram clusters voice patterns and splits the transcript into speaker-labelled segments (e.g., `Speaker 0`, `Speaker 1`). ScribeFlow stores these index numbers and allows users to rename them to actual human identities later.

### How WER is Calculated

Word Error Rate (WER) measures transcription accuracy. It compares a reference (human-verified) transcript with the AI transcript. The formula is:
**(Substitutions + Deletions + Insertions) / Total Reference Words**
Our verifier cleans up non-spoken labels (like `YOU:`) before calculating to ensure an honest score.

### Summary and Action Item Extraction

After transcription, the backend sends the full text to Gemini. Gemini analyzes the transcript and populates our predefined JSON schema. ScribeFlow maps these JSON results into relational database tables (Action Items, Topics, Summaries).

### Avoiding Hallucinations

Hallucination occurs when an AI invents information. ScribeFlow avoids this in two ways:

1. **Strict Schema:** We force Gemini to extract specific fields (task, owner, deadline).
2. **Evidence Links:** Gemini must provide the original `segmentId` that justifies the action item or summary point.

### Evidence Timestamps

Because we store exact word and segment timestamps from Deepgram, when Gemini cites a `segmentId`, the frontend can create a deep-link. Clicking an action item's "target" icon jumps the user to the exact moment in the transcript and highlights the source audio segment.

## Semantic Search & RAG

### What are Embeddings?

Embeddings are high-dimensional arrays of numbers (vectors) that represent the _meaning_ of text. Similar concepts (e.g., "budget" and "financials") will have vectors that are mathematically close to each other.

### What is pgvector?

`pgvector` is a PostgreSQL extension that allows the database to store embeddings and perform fast similarity searches (using mathematical distance functions like Cosine Similarity) directly within SQL queries.

### How Semantic Search Works (RAG)

1. **Chunking:** The transcript is split into smaller, overlapping chunks.
2. **Embedding:** We send these chunks to Gemini to get vector embeddings.
3. **Indexing:** We save these chunks and vectors into Supabase.
4. **Searching:** When a user searches, we embed their query, use an RPC (`match_meeting_chunks`) in pgvector to find the closest chunks, and return the relevant transcript segments.

## Security & Reliability

### Private Storage & Backend-Only Keys

Audio files are stored in private Supabase buckets. They are never publicly accessible. Deepgram reads the audio via a temporary, short-lived signed URL. Furthermore, provider keys (`DEEPGRAM_API_KEY`, `GEMINI_API_KEY`) stay strictly on the backend. The frontend only receives safe, public data.

### Idempotency

Idempotency ensures that running the same operation multiple times doesn't cause duplicate data or corrupted states. For example, our `replace_meeting_transcription` database function uses an atomic SQL transaction: it deletes old speakers and segments, inserts the new ones, and updates the meeting status all at once. If it fails halfway, the database rolls back cleanly.

## Limitations & Future Improvements

### Current Limitations

- AI extraction can still occasionally misinterpret highly ambiguous deadlines or unassigned tasks.
- Large meetings (>2 hours) require heavy token usage, which could hit Gemini API rate limits on free tiers.
- Cross-meeting RAG chat is not fully implemented (currently supports global semantic search, but not conversational synthesis).

### Future Improvements

- **Live Recording WebSockets:** Stream audio directly from the browser to Deepgram for real-time transcription.
- **Cross-Meeting Synthesis:** Allow users to chat with the system to summarize decisions across _all_ meetings in a specific project.
- **User Authentication:** Add Supabase Auth to isolate workspaces and restrict meeting access by user.
